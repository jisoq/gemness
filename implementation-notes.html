Gemness Workspace Allow Policy Implementation Notes

Last updated: 2026-05-21 KST. This file is maintained during implementation whenever the realized design clarifies, differs from, or extends the goal.

설계 결정

  • Branch작업은 새 브랜치 codex/workspace-allow-policy에서 시작했다.
  • Policy shape기존 tuple[Path] allowed-root 계산만으로는 nested untrusted override를 표현하기 어렵기 때문에, WorkspaceDecision과 WorkspaceAccessError 중심의 정책 판단 표면을 추가한다.
  • Trust sourceCodex trust는 CODEX_HOME/config.toml 또는 ~/.codex/config.toml의 [projects] table에서 읽고, Python 3.11+ 표준 tomllib만 사용한다.
  • Closest matchCodex project trust 판정은 candidate와 같거나 candidate의 parent인 entry 중 가장 긴 path를 선택한다. 따라서 child untrusted가 parent trusted를 명확히 덮어쓴다.
  • Invalid trust가장 가까운 Codex project entry의 trust_level이 없거나 지원하지 않는 값이면 그 entry를 허용 근거로 쓰지 않는다. 더 상위 trusted parent로 fallback하지 않고 absent로 판단해 warning을 남기는 쪽이 closest-match 규칙과 안전성에 더 맞다고 해석했다.
  • Health statusantigravity_health는 workspace policy deny를 서버 오류로 보지 않고 status="warning"과 workspace diagnostics로 반환한다. 실행 tool만 status="error"로 차단한다.

편차

  • trust-workspacegemness trust-workspace <path>는 선택 사항으로 해석해 이번 1차 패치에는 포함하지 않았다. Codex config TOML을 안전하게 보존하면서 쓰는 별도 writer가 필요하고, 핵심 UX는 Codex trusted projects 자동 인식과 skill의 cwd 전달로 해결된다.

트레이드오프

  • Safety명시 GEMNESS_ALLOWED_ROOTS가 있으면 Codex trusted projects를 완전히 무시하는 strict mode를 택한다. 편의성은 줄지만, 사용자가 보안 경계를 명시한 경우 그 의도를 우선한다.
  • Compatibility아무 정책 근거가 없는 arbitrary cwd 자동 허용은 기본적으로 막고, 필요 시 GEMNESS_ALLOW_UNTRUSTED_CWD_FALLBACK=true opt-in으로만 기존 동작을 허용하는 방향으로 구현한다.
  • Follow-upfollow_up_antigravity는 cwd 인자가 없지만 실제 agy 실행을 만들기 때문에 parent session의 project_root를 다시 policy 검증한다. parent root가 사라졌으면 기본 cwd policy로 fallback하되, 그 역시 신뢰 근거가 없으면 차단한다.

미결 질문

  • CLI trust후속으로 gemness trust-workspace <path>를 추가한다면 Codex config의 기존 서식/주석 보존 수준을 어디까지 보장할지 결정해야 한다.
  • Malformed trustCodex config의 project entry가 문자열 path가 아니거나 trust_level이 예상 값이 아닐 때 warning payload의 정확한 표현을 테스트와 함께 확정한다.
