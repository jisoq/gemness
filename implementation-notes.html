Gemness Antigravity CLI Migration Notes

실시간 기록 파일. 2026-05-20 현재 Antigravity CLI 전환 조사 중 발견한 설계 결정, 편차, 트레이드오프, 미결 질문을 갱신한다.

설계 결정

  • implementation-notes.html은 별도 경로가 지정되지 않았으므로 저장소 루트에 유지한다.
  • Antigravity IDE와 Antigravity CLI는 별도 표면으로 취급한다. 이번 조사는 agy 기반 Antigravity CLI 문서와 Gemini CLI 마이그레이션 문서만 기준으로 삼고, IDE 전용 기능은 구현 근거로 사용하지 않는다.
  • 공식 문서는 SPA HTML이 아니라 https://antigravity.google/assets/docs/cli/*.md의 Markdown 원문에서 확인한다. 현재 확인한 핵심 문서는 gcli-migration.md, cli-getting-started.md, cli-using.md, cli-features.md, cli-overview.md이다.
  • Antigravity CLI 설치 명령은 문서상 Windows PowerShell 기준 irm https://antigravity.google/cli/install.ps1 | iex, 실행 명령은 agy로 보인다.
  • 현재 Gemness는 사용자-facing 이름뿐 아니라 실행 환경 변수와 runner 구현 전반에서 gemini/Gemini CLI를 canonical backend로 가정한다. 마이그레이션은 단순 문자열 교체보다 CLI backend 추상화가 안전하다.
  • 현재 MCP 서버 자체는 Codex용 MCP 서버이므로 Antigravity CLI의 MCP 설정 파일 변경은 Gemness runtime보다는 설치/문서/부트스트랩 표면의 추가 타깃으로 다룬다.
  • Gemness MCP를 통해 Gemini에 자문을 요청했다. 자문 결과도 정적 설정/문서 변경은 쉽지만, agy headless 플래그와 stream JSON 스키마 검증이 고위험 선행조건이라는 판단이었다. 이 판단은 현재 조사 결과와 일치하므로 수용한다.
  • Antigravity CLI 설치 스크립트는 manifest와 SHA512 검증 후 %LOCALAPPDATA%\agy\bin\agy.exe로 바이너리를 배치하고, 후속 setup으로 agy.exe install을 실행한다. 설치 후 이 경로를 우선 검증 대상으로 사용한다.
  • agy 1.0.0 설치가 성공했다. 현재 터미널 PATH에는 아직 반영되지 않았지만, 직접 경로 C:\Users\User\AppData\Local\agy\bin\agy.exe로 실행 가능하다.
  • 실제 agy --help 기준 non-interactive 실행은 -p/--print/--prompt 플래그를 사용한다. 대화 재개는 --continue 또는 --conversation <id> 형태다.

편차

  • Antigravity CLI는 Gemini CLI와 기능 대부분을 지원하지만 100% parity는 아니라고 명시되어 있다. 특히 headless 실행 플래그의 정확한 parity는 문서만으로 확정되지 않았다.
  • 현재 Gemness runner는 -p, --output-format stream-json, --approval-mode plan, --skip-trust, --session-id, --resume 같은 Gemini CLI 플래그와 stream JSON 이벤트 형식을 직접 가정한다.
  • Antigravity CLI 문서는 /resume UI와 종료 시 출력되는 resume command를 설명하지만, 비대화형 --resume/--session-id/-p/--output-format stream-json 지원 여부를 아직 공식 문서에서 확인하지 못했다.
  • Antigravity CLI의 설정 경로는 ~/.gemini/antigravity-cli/settings.json, MCP 설정은 ~/.gemini/antigravity-cli/mcp_config.json 또는 .agents/mcp_config.json이다. 기존 Gemini CLI의 ~/.gemini/settings.json/.gemini/settings.json과 다르다.
  • Antigravity CLI remote MCP 설정은 url/httpUrl 대신 serverUrl을 사용한다. Gemness의 현재 Codex TOML 부트스트랩과 별도 포맷이다.
  • 로컬 PATH에서는 agy를 찾지 못했고, C:\Users\User\AppData\Local\Programs\Antigravity\bin\antigravity.cmd만 발견했다. 사용자가 IDE와 CLI를 구분하라고 했으므로 이 실행기는 CLI 호환성 검증에 사용하지 않는다.
  • 코드 검색 결과 Gemini CLI 결합부는 src/gemness/config.py, src/gemness/runner.py, src/gemness/tools.py, src/gemness/cli.py, src/gemness/codex_install.py, src/gemness/web.py, README/INSTALL/docs/tests 전반에 분포한다.
  • 실제 agy 1.0.0은 --output-format, --resume, --session-id, --model, --reasoning-effort를 정의하지 않은 플래그로 거부한다.
  • agy -p print mode는 현재 인증이 없어 모델 응답을 받지 못했다. 로그에는 You are not logged into Antigravity와 model=""이 기록된다.
  • agy -p 실행 로그에서 프로젝트 discovery가 .antigravitycli symlink 생성 권한 부족으로 실패했다. CLI는 계속 진행하지만 workspaceDirs=[]로 backend를 만들기 때문에 프로젝트 컨텍스트 누락 가능성이 있다.
  • 사용자 화면의 d3dcompiler_47.dll 교체 오류는 그래픽 DLL을 건드리는 Windows GUI 설치/업데이트로 보이며, CLI 바이너리 설치 자체는 이미 완료되었다. IDE/CLI를 혼동하지 않기 위해 이 GUI 오류는 CLI 검증 blocker로 보지 않는다.

트레이드오프

  • 기본 명령을 즉시 gemini에서 agy로 바꾸면 기존 Gemini CLI 사용자가 깨질 수 있다. 단기적으로는 GEMNESS_COMMAND 또는 새 backend 설정으로 양쪽을 선택 가능하게 두고, 감지/health_check에서 backend 이름과 플래그 지원을 분리하는 편이 안전하다.
  • 기존 환경 변수명을 모두 GEMNESS_GEMINI_*에서 GEMNESS_ANTIGRAVITY_*로 바꾸면 호환성이 크게 깨진다. 호환 alias를 유지하면서 내부 이름만 중립화하는 방식이 더 reviewable하다.
  • Antigravity CLI headless 플래그가 Gemini CLI와 다르면 Observer의 실시간 스트림, JSON envelope 합성, native resume 로직을 backend별 adapter로 나누어야 한다. 구현량은 늘지만 런타임 실패를 명확히 격리할 수 있다.
  • agy에는 현재 stream JSON 출력 플래그가 없으므로, Antigravity backend는 초기 구현에서 실시간 token delta를 포기하고 print-mode 최종 stdout을 캡처하는 방식이 더 현실적이다. Observer에는 streaming unavailable 상태를 명시해야 한다.
  • Antigravity CLI MCP 설정 지원을 Gemness에 추가하면 설치 대상이 Codex와 Antigravity CLI 두 개로 늘어난다. 문서/부트스트랩 표면은 명확해지지만, IDE 설정과 CLI 설정을 혼동하지 않도록 파일명을 엄격히 분리해야 한다.
  • 이벤트 이름을 gemini.started 같은 기존 명칭으로 유지하면 Observer/테스트 변경은 작지만 새 backend 의미가 흐려진다. llm.started 같은 중립 이벤트를 도입하면 장기적으로 깔끔하지만 transcript 하위 호환 마이그레이션이 필요하다.
  • Windows에서 Antigravity CLI 프로젝트 symlink 생성을 요구한다면 일반 사용자 권한에서 workspace discovery가 약해질 수 있다. 해결책은 개발자 모드/권한 안내, symlink 없이 동작하는 fallback 확인, 또는 Gemness fallback prompt에 cwd/context를 명시적으로 싣는 방식이다.

미결 질문

  • Antigravity CLI가 향후 공개할 수 있는 non-interactive structured output 또는 event stream 방식이 있는지 확인해야 한다. 현재 agy 1.0.0 help/flag parser에서는 확인되지 않는다.
  • Gemness MCP의 공개 명칭을 계속 "Gemness/Gemini advisory"로 유지할지, Antigravity backend를 지원하는 중립 명칭과 문구로 바꿀지 결정이 필요하다.
  • Gemness가 Antigravity CLI용 .agents/mcp_config.json 생성기를 제공해야 하는지, 아니면 Codex MCP 설치만 유지하고 문서로 수동 설정을 안내할지 결정이 필요하다.
  • Antigravity CLI를 설치해도 기존 gemini 명령과 병행 지원해야 하는지, 아니면 Gemness의 기본 backend를 agy로 전환해야 하는지 정책 결정이 필요하다. 현재 실물 검증으로는 병행 adapter가 더 안전하다.
  • 사용자 요청에 따라 기본 모델은 3.5 flash (high)로 변경되어야 한다. 정확한 CLI model identifier와 reasoning/high 옵션 표현은 agy --help 및 관련 설정 출력으로 확인해야 한다.
  • 공식 Models 문서는 reasoning model 목록에 Gemini 3.5 Flash를 표시하지만 Gemini 3.5 Flash (high)라는 표기는 별도로 보이지 않는다. 바이너리 내부에는 MODEL_GOOGLE_GEMINI_RIFTRUNNER_THINKING_HIGH 같은 internal enum이 보이지만 공개 설정값으로 사용할 수 있는지는 미확인이다.
  • 인증 전에는 FetchAvailableModels가 실패하므로 Gemini 3.5 Flash의 공개 설정 identifier와 high variant 여부를 실물 API로 확정할 수 없다.
